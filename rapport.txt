1. PROJECT INTRODUCTION
================================================================================

MarketingERP is a full-stack Enterprise Resource Planning platform designed
specifically for marketing and creative agencies. It centralises project
management, task tracking, client relations, analytics, budget control, and
AI-powered reporting into a single unified web application.

The platform serves two distinct portals:

  Internal Dashboard — used by agency staff across five role tiers:
    Admin, Marketing Manager, Marketing Agent, Production Manager,
    Production Agent. Each role has carefully scoped permissions so
    users only access data relevant to their responsibilities.

  Client Portal — a separate, publicly accessible interface where
    clients can track their active projects, review and approve tasks
    submitted for their sign-off, view shared reports, and communicate
    feedback without needing an internal account.

The system is built as a monorepo (npm workspaces) containing two
applications: apps/api (NestJS REST + WebSocket server) and apps/web
(Next.js 16 SPA). They communicate over HTTP/REST and Socket.io in
real-time.

================================================================================
2. OBJECTIVES & SCOPE
================================================================================

Primary objectives:
  - Provide a centralised workspace for managing the full lifecycle of
    agency projects — from client acquisition to final delivery.
  - Automate administrative tasks: AI-generated performance reports,
    real-time notifications, and background job scheduling.
  - Offer clients full transparency via a dedicated portal with review
    and approval workflows.
  - Enforce strict role-based access control across all resources.
  - Maintain a complete, immutable audit trail of every sensitive action.

Out of scope:
  - Payment processing / invoicing
  - External calendar sync (Google, Outlook)
  - Mobile native applications

3. TECHNOLOGY STACK
================================================================================

  Backend (apps/api)
  ------------------
  Runtime         : Node.js 20 LTS
  Framework       : NestJS 11
  Language        : TypeScript 5
  ORM             : Prisma 6 + PostgreSQL 16
  Auth            : JWT (access 15 min, refresh 7 days), bcrypt
  Queues          : BullMQ + Redis 7
  Real-time       : Socket.io + @nestjs/websockets
  AI              : Anthropic Claude SDK ^0.95 (model: claude-sonnet-4-6)
  External APIs   : Meta Graph API (Ads sync), Resend (email), AWS S3 (files)
  Validation      : class-validator + class-transformer
  Logging/Audit   : Custom AuditLogService (DB-persisted)

  Frontend (apps/web)
  -------------------
  Framework       : Next.js 16.2 (App Router)
  Language        : TypeScript 5
  Styling         : CSS Variables + Tailwind utilities
  Animation       : Framer Motion
  State           : Zustand
  Icons           : Lucide React
  Fonts           : Syne (headings) + DM Mono (labels/data)

  Infrastructure
  --------------
  Database        : PostgreSQL 16
  Cache / Queues  : Redis 7
  Process manager : pm2 (production)
  Monorepo        : npm workspaces

5.4  ENTITY-RELATIONSHIP DIAGRAM
--------------------------------------------------------------------------------

  users
  -----
  id (PK), email (UQ), password_hash, full_name, role, department,
  is_active, created_at, updated_at

  clients
  -------
  id (PK), name, contact_email (UQ), phone, industry, address,
  portal_password_hash, portal_token, created_at, updated_at

  projects
  --------
  id (PK), name, objective, status, start_date, end_date, budget_total,
  currency, meta_ads_account_id, client_id (FK->clients),
  created_by_id (FK->users), created_at, updated_at

  project_users
  -------------
  project_id (FK->projects), user_id (FK->users), created_at
  (composite unique key on [project_id, user_id] — no separate role column)

  tasks
  -----
  id (PK), title, description, status, priority, project_id (FK->projects),
  created_by_id (FK->users), due_date, revision_count,
  client_approval (none|pending|client_approved|client_rejected),
  client_rejection_comment, client_reviewed_by (FK->client_users),
  client_reviewed_at, created_at, updated_at

  task_assignees
  --------------
  task_id (FK->tasks), user_id (FK->users)
  (many-to-many — a task can have multiple assignees, not a single FK)

  task_comments
  -------------
  id (PK), task_id (FK->tasks), user_id (FK->users),
  content, created_at

  task_attachments
  ----------------
  id (PK), task_id (FK->tasks), s3_key, file_name, mime_type,
  size_bytes, uploaded_by_id (FK->users), created_at

  reports
  -------
  id (PK), project_id (FK->projects), period_start, period_end,
  status (generating|ready|failed), executive_summary, insights (JSON),
  recommendations (JSON), is_shared, created_at, updated_at

  budget_entries
  --------------
  id (PK), project_id (FK->projects), amount, category, description,
  entry_date, created_by_id (FK->users), created_at

  notifications
  -------------
  id (PK), user_id (FK->users), type, message, link,
  is_read, created_at

  audit_log
  ---------
  id (PK), user_id (FK->users, nullable), action, entity, entity_id,
  ip_address, meta (JSON), created_at

  analytics_records
  ------------------
  id (PK), project_id (FK->projects), date, impressions, clicks, spend,
  reach, conversions, ctr, cpc, roas (synced from Meta Graph API)

  sync_jobs
  ---------
  id (PK), project_id (FK->projects), status (running|succeeded|failed),
  started_at, finished_at, error_message

  ai_insights
  -----------
  id (PK), project_id (FK->projects), summary, insights (JSON),
  recommendations (JSON), created_at

  Note: no "deliverables" table and no "chat_messages" table exist in the
  schema. Client review/approval lives directly on tasks (client_approval
  field). The AI chatbot does not persist conversation history server-side.

  Relationships:
    clients        ||--o{ projects        (one client, many projects)
    users          ||--o{ projects        (created_by)
    projects       ||--o{ project_users   (many members)
    users          ||--o{ project_users   (member of many projects)
    projects       ||--o{ tasks
    tasks          ||--o{ task_assignees  (many-to-many with users)
    tasks          ||--o{ task_comments
    tasks          ||--o{ task_attachments
    client_users   ||--o{ tasks           (client_reviewed_by)
    projects       ||--o{ reports
    projects       ||--o{ budget_entries
    projects       ||--o{ analytics_records
    projects       ||--o{ sync_jobs
    projects       ||--o{ ai_insights
    users          ||--o{ notifications
    users          ||--o{ audit_log

12. SECURITY DESIGN
================================================================================

  Authentication:
    - Passwords hashed with bcrypt (salt rounds: 12)
    - Access tokens: JWT HS256, 15-minute TTL
    - Refresh tokens: JWT HS256, 7-day TTL, hash stored in DB
    - Refresh tokens rotated on each use (old hash invalidated)
    - Portal clients use separate JWT (type: 'portal') to prevent
      cross-portal access

  Authorisation:
    - JwtAuthGuard applied globally (all routes protected by default)
    - @Public() decorator allows explicit opt-out (login, refresh)
    - RolesGuard checks @Roles() decorator on each endpoint
    - Service-layer scoping: agents only query their own assigned data
    - Project membership verified before returning project details

  Data Protection:
    - Passwords and portal password hashes never returned in API responses
    - Refresh token hashes stored (never the raw token)
    - .env file gitignored; .env.example contains no real secrets
    - Prisma prevents raw SQL injection by design
    - File uploads validated for MIME type and size

  Audit Trail:
    - Every create, update, delete action logged to audit_log table
    - Audit logs are append-only (no delete endpoint)
    - Logs include: user_id, action, entity type, entity ID, metadata JSON

  Transport:
    - HTTPS enforced in production
    - WebSocket connections require valid JWT on handshake
    - CORS configured to allow only trusted origins

13. CONCLUSION
================================================================================

The MarketingERP Platform represents a complete, production-ready full-stack
ERP solution tailored for marketing and creative agencies. Over five two-week
sprints, the system evolved from a bare authentication skeleton to a fully
featured platform supporting:

  - Five internal roles with fine-grained, scope-based access control
  - Full project lifecycle management from client onboarding to delivery
  - Real-time collaboration via WebSockets and push notifications
  - AI-powered report generation and an AI chatbot assistant using
    Anthropic Claude Sonnet 4.6
  - Automated Meta Ads performance data sync via the Facebook Graph API
  - Transactional email notifications via Resend
  - A dedicated client portal with task review/approval workflows
  - An immutable audit trail for compliance and governance
  - A clean, accessible UI with light/dark theme support

The architecture — NestJS + Prisma + PostgreSQL + Redis + BullMQ + Next.js —
provides a scalable foundation. Background job queues decouple AI processing
and Meta Ads sync from the request cycle. WebSocket rooms enable targeted
real-time push without broadcasting unnecessary data. The monorepo structure
keeps shared configuration and types close to both applications while
keeping build processes independent.

22 of the 25 user stories from the product backlog have been delivered.
The three remaining items — payment/invoicing integration, external
calendar sync, and a native mobile app — were explicitly scoped out and
represent the natural next phase of the product.
